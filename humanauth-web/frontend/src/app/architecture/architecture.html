HumanAuth System Architecture

Face and hand recognition to verify humans vs. bots.

HumanAuth is a biometric authentication system that uses computer vision to distinguish real humans from automated bots or spoofing attempts. Unlike traditional CAPTCHAs, it analyzes natural human movements and responses to interactive challenges.

System Overview

HumanAuth works through a three-step process: the frontend captures video frames from the user's camera, the backend processes these frames using MediaPipe's machine learning models, and then verifies the user's identity through interactive challenges that require natural human responses.

⚙️ Backend Architecture

🔧

Flask Server (app.py)

The Flask server provides REST API endpoints that handle all communication with the frontend. It manages user sessions, processes incoming video frames, and coordinates the authentication workflow. The server validates requests, ensures proper session management, and implements security measures like rate limiting and API key validation.

🤖

Human Authentication Engine

The core of HumanAuth is its authentication engine that uses MediaPipe's machine learning models to detect and analyze facial landmarks (468 points) and hand gestures (21 points per hand). The engine implements sophisticated anti-spoofing techniques that detect micro-movements, analyze blink patterns, and verify 3D spatial consistency to prevent attacks using photos, videos, or deepfakes.

🎨 Frontend Architecture

Angular Application

The frontend is built with Angular and consists of three main components:

  • Auth Page Component: Provides the user interface for the authentication process, displaying camera feed, challenge instructions, and authentication status.
  • Camera Service: Handles camera initialization, video stream capture, and frame extraction. It manages permissions, camera selection, and optimizes frame rate for processing.
  • Auth Service: Manages communication with the backend API, handles session creation, frame submission, and processes authentication responses and challenges.

🌐 API Communication

Key REST API Endpoints

The HumanAuth system uses a RESTful API architecture for communication between the frontend and backend. All data is exchanged in JSON format, and authentication is secured using API keys.

GET
/api/config

Retrieves API configuration including supported features, challenge types, and processing parameters. This is the first endpoint called during initialization.

POST
/api/sessions

Creates a new authentication session with a unique session ID. The session maintains state throughout the authentication process and tracks challenge progress.

POST
/api/sessions/{session_id}/process

Processes a video frame, performs face and hand detection, evaluates ongoing challenges, and returns the next steps in the authentication flow. This endpoint is called repeatedly during the authentication process.

Security Considerations

HumanAuth implements multiple layers of security to prevent various types of spoofing attacks while maintaining user privacy. The system is designed to detect sophisticated attempts to bypass authentication, including the use of photos, videos, deepfakes, and 3D masks.

Anti-Spoofing Defense Layers

Physical Presence Detection
  • Micro-movement analysis: Detects subtle natural movements that occur even when a person is trying to remain still
  • Blink pattern detection: Analyzes natural eye blink patterns that are difficult to simulate
  • 3D consistency checks: Verifies that facial features maintain proper spatial relationships as the head moves
Interactive Challenge System
  • Randomized challenges: Unpredictable sequence of actions that cannot be pre-recorded
  • Multi-modal verification: Combines face and hand detection for stronger security
  • Temporal analysis: Verifies that responses occur within expected timeframes

Privacy Protections

Privacy-First Design

No biometric storage: All biometric data is processed in memory and immediately discarded

Session isolation: Each authentication session is isolated and data is not shared between sessions

Minimal data collection: Only collects the data necessary for authentication, not for identification
